# RPF Relational Viability Process Framework

## 0. Purpose

This document defines the deeper substrate beneath RPF's dramatic conflict framework.

The simulator should not begin from:

```text
personality
emotion
desire
relationship score
dramatic conflict
plot need
```

It should begin from:

```text
constrained relational viability.
```

The core claim is:

```text
Human-like drama emerges when embodied, relational, symbolic, and material processes must continue under constraints that make continuation costly, distorted, unstable, or impossible.
```

Therefore, RPF does not primarily simulate conflict.

RPF simulates:

```text
how relational processes preserve, deform, fail, repair, and stabilize themselves under constraint.
```

Conflict is a derived visibility condition. It appears when viability requirements are blocked, forced into contradiction, observed by another process, and stabilized into memory, identity, relation, or fate.

---

## 1. Ontological Reversal

The dramatic layer may speak in terms of conflict, tension, recognition, betrayal, resentment, sacrifice, or fate.

The causal layer must speak in terms of:

```text
field
constraint
viability requirement
affordance width
adaptive response
deformation
observation
recognition / misrecognition
stabilization
future constraint
derived view
```

This means:

```text
not: a person has a desire, then conflict occurs.
but: a process has a viability requirement under constraint, then adaptation or deformation occurs.
```

`Person`, `Relationship`, `Personality`, `Emotion`, and `Dramatic Conflict` remain derived phenomena.

They may be rendered, inspected, named, and aggregated.

They must not be primitive causal objects.

---

## 2. Core Unit: Process Under Constraint

The deepest practical unit is:

```text
ProcessUnderConstraint
```

It is not a person. It is a local relational process position that is currently exposed to field pressures and must preserve some form of viability.

```text
ProcessUnderConstraint
- process_id
- field_position
- active_constraints
- active_viability_requirements
- affordance_width
- response_cost
- deformation_risk
- recognition_dependency
- repair_access
- memory_load
- future_constraint_load
- evidence_refs
```

This unit should answer:

```text
What must continue?
Under what constraint?
What responses are available?
What is the cost of direct response?
What deformation becomes likely?
Who can observe or misrecognize the deformation?
What future possibilities are narrowed?
```

## 3. Layered Causal Stack

The mature RPF kernel should be organized as a layered aggregation system.

```text
Layer 0: Field Configuration
Layer 1: Constraint Activation
Layer 2: Viability Requirement Activation
Layer 3: Affordance Narrowing
Layer 4: Adaptive Response
Layer 5: Deformation / Inhibition / Direct Exercise
Layer 6: Observation and Interpretation
Layer 7: Relevance Landscape Update
Layer 8: Second-Order Expectation Sedimentation
Layer 9: Account Pressure Projection
Layer 10: Normative Pressure Projection
Layer 11: Interaction Frame Definition
Layer 12: Relation-Generated Positioning
Layer 13: Recognition / Misrecognition
Layer 14: Repair / Avoidance / Escalation
Layer 15: Stabilization
Layer 16: Memory Reconstruction
Layer 17: Future Constraint Update
Layer 18: Environment Sedimentation
Layer 19: Process Disposition Sedimentation
Layer 20: Relation Sedimentation
Layer 21: Binding Evolution
Layer 22: Derived Person / Relationship / Fate Views
Layer 23: Dramatic Conflict and Narrative Rendering
```

No higher layer may authorize a lower layer.

For example:

```text
invalid: this is a betrayal scene, so action should become cruel.
valid: prior recognition refusal plus public role pressure narrows repair affordances, increasing defensive deformation and future interpretation bias; the rendered scene may be read as betrayal.
```

## 4. Viability Requirements

A `ViabilityRequirement` is a process-level requirement that must be satisfied, deferred, displaced, or deformed for the process to continue.

It is not a goal owned by a person.

Initial requirement classes:

```text
ViabilityRequirement
- bodily_continuation
- resource_access
- spatial_access
- temporal_continuation
- relation_continuation
- role_continuation
- face_continuation
- recognition_access
- autonomy_preservation
- dependency_management
- secrecy_maintenance
- truth_integration
- memory_integration
- identity_continuity
- care_delivery
- exit_availability
- repair_availability
```

Each requirement must specify:

```text
- source
- activation_condition
- target_process optional
- urgency
- negotiability
- minimum_satisfaction_condition
- failure_cost
- deformation_tendency
- evidence_refs
```

A requirement is invalid if it can only be described as:

```text
the character wants X.
```

It becomes valid when it can be described as:

```text
this process cannot continue coherently unless X is satisfied, deferred, displaced, or transformed.
```

---

## 5. Constraint Activation

A `Constraint` is not an obstacle in a story.

It is a field condition that changes what kinds of continuation are possible.

Valid constraints must define:

```text
- source layer
- activation condition
- affected process
- affected viability requirement
- intensity
- duration policy
- decay or reinforcement rule
- reversibility
- downstream effects
- evidence_refs
```

Constraint types:

```text
material
bodily
spatial
temporal
institutional
economic
informational
symbolic
historical
relational
public_audience
memory
classification
```

The same action may be available under one constraint field and impossible under another.

Example:

```text
Apology is materially possible.
Apology is face-threatening in public.
Apology is institutionally dangerous at work.
Apology is identity-incoherent after repeated denial.
```

Therefore, RPF should never ask only:

```text
what action does the process choose?
```

It should ask:

```text
what kinds of continuation are currently affordable?
```

---

## 6. Adaptive Response and Deformation

When a viability requirement can be directly satisfied, the process may perform a direct adaptive response.

When direct satisfaction is blocked, the response may become deformed:

```text
recognition need -> accusation
repair need -> practical help
exit need -> silence
care need -> control
autonomy need -> refusal
dependency need -> resentment
truth integration -> joke
face preservation -> public performance
fear -> politeness
desire -> irritation
```

Deformation is not random expressive flavor.

It is the shape a blocked requirement takes when it must still move through a constrained field.

Every deformation should cite:

```text
- blocked requirement
- blocking constraint
- chosen visible form
- deformation distance
- ambiguity
- observer risk
- expected recognition failure modes
- evidence_refs
```

## 8. Derived Drama

Dramatic conflict is a derived aggregate, not a base entity.

One possible derived formula:

```text
dramatic_tension =
  high viability_pressure
+ low affordance_width
+ high response_cost
+ high deformation_distance
+ low recognition_availability
+ low repair_accessibility
+ high stabilization_strength
+ high future_constraint_load
```

This formula does not create conflict.

It identifies when a process history has become narratively visible as conflict.

The LLM may render this visibility, but it may not authorize it.

## 9. Runtime Kernel

The deeper runtime loop should be:

```text
1. Load state and recent history.
2. Evaluate field configuration.
3. Activate constraints.
4. Activate viability requirements.
5. Measure affordance width.
6. Evaluate direct satisfaction routes.
7. Detect blocked or contradictory requirements.
8. Select adaptive response.
9. Determine deformation, inhibition, or direct exercise.
10. Emit observable signal.
11. Evaluate observation and interpretation.
12. Evaluate recognition or misrecognition.
13. Evaluate repair, avoidance, escalation, or delay.
14. Stabilize repeated patterns.
15. Reconstruct memory.
16. Update future constraints.
17. Recompute derived person, relationship, fate, and drama views.
18. Persist event stream and traces.
```

The existing RPF dramatic conflict framework should be implemented as a derived subsystem over steps 3 through 17.

## 10. Simulation Upgrade Priority

The next kernel upgrade should not begin with "make stories more dramatic."

It should begin with:

```text
Trace-Only Relational Viability Kernel
```

First implementation scope:

```text
1. Add trace-only ViabilityRequirement events.
2. Add trace-only ConstraintActivation events with affected requirements.
3. Add AffordanceWidth measurement.
4. Add DeformationDistance measurement for action/expression events.
5. Derive DramaticConflict indicators from those traces.
6. Ensure existing scenarios produce identical causal outputs at first.
```

Only after the trace layer is stable should it gain limited causal authority.

Recommended authority schedule:

```text
Stage 0: trace only.
         implemented as ConstraintActivationEvent, ViabilityRequirementEvent, AffordanceWidthEvent, DeformationTraceEvent, and DerivedDramaticTensionEvent.

Stage 1: recognition evaluation may cite viability/blockage evidence.
         implemented as bounded recognition-score adjustments plus explicit evidence refs.

Stage 2: expression selection may use deformation pressure.
         implemented as bounded expression-score adjustments from pre-response viability events.

Stage 3: action selection may use affordance width and response cost.
         implemented as very small action-score adjustments from pre-response viability events.

Stage 4: scheduler may use accumulated viability pressure.
         implemented as state-only viability rhythm preview and bounded tick-type score bias.
```

Stage 1 must remain bounded:

```text
- it may add small score adjustments to recognition outcomes.
- it must include viability evidence in RecognitionEvent payloads.
- RecognitionEvent causal_refs must include the viability events it consumed.
- it must not create scenes, force plot outcomes, mutate derived views, or bypass action/expression events.
```

Stage 2 must remain bounded:

```text
- it may bias expression form using viability pressure, direct response cost, affordance narrowing, and face constraint.
- it must consume only pre-response viability evidence.
- ExpressionSelectionEvent causal_refs must include the viability events it consumed.
- it must not change the selected action, create a new action, or bypass observation and recognition.
```

Stage 3 must remain narrower than Stage 2:

```text
- it may bias action mode using viability pressure, direct response cost, affordance narrowing, and face constraint.
- it must consume only pre-response viability evidence.
- ActionSelectionEvent causal_refs must include the viability events it consumed.
- it must not bypass AffordanceSelectionEvent.
- it must not create actions outside the available candidate set.
- it must preserve benchmark-level multi-attractor diversity.
```

Stage 4 must remain temporal, not dramaturgical:

```text
- it may compute a state-only scheduler preview before tick events exist.
- it may add tiny rhythm bias to scene and micro-interaction scores.
- it must record viability_rhythm diagnostics.
- it must not emit viability events before the tick begins.
- it must not force a scene because "drama is needed".
- it must preserve latent, micro-interaction, and scene tick diversity across benchmark scenarios.
```

Stage 5 makes history constrain the future:

```text
- it derives FutureConstraintEvent from irreversibility records, operative classifications, and reconstructed memories.
- it converts those traces into bounded future_constraint ConstraintActivationEvent entries on later ticks.
- it must cite the source event, record, classification, or memory that produced the constraint.
- it must not invent future consequences or skip ordinary action, expression, recognition, memory, or irreversibility pathways.
```

Stage 6 lets future constraints bias pattern recurrence:

```text
- RPP activation may consume FutureConstraintEvent as a small eligibility-score component.
- RPPActivationEvent eligibility_evidence and causal_refs must include consumed FutureConstraintEvent ids.
- future constraints must not directly activate an RPP without the RPP's ordinary field, signal, memory, or recognition gates.
- this is the mechanism by which history becomes tendency rather than a stored label.
```

Stage 7 lets future constraints shape memory reconstruction:

```text
- Memory reconstruction may consume FutureConstraintEvent as bounded salience and confidence modulation.
- FutureConstraintEvent may only modulate an already produced memory candidate.
- It must not create a MemoryReconstructionEvent without failed recognition, operative classification, or irreversibility.
- MemoryReconstructionEvent evidence and causal_refs must include consumed FutureConstraintEvent ids.
- this is the mechanism by which history filters what later becomes memorable.
```

Stage 8 lets relation history sediment into the field:

```text
- Environment sedimentation may consume RPPActivationEvent, FutureConstraintEvent, MemoryReconstructionEvent, IrreversibilityEvent, and prior-tick RelationSedimentationEvent.
- It may update bounded material, spatial, and audience field pressures.
- It may create EnactedMicroWorldEvent when a space, object, routine, or audience position gains durable relational meaning.
- It must not create new interpersonal outcomes by itself.
- Sedimented field pressures must decay unless reinforced by later interaction.
- sedimented field pressure can later be consumed by viability evaluation and situated affordance selection.
- Relation sediment may attach unsettled recognition debt to material prompts, repair narrowing to avoidance paths, public definition to imagined audiences, and fate load to memory-saturated space.
```

Stage 9 lets evidence sediment into process dispositions:

```text
- DispositionSedimentationEvent may consume MemoryReconstructionEvent, FieldUpdateEvent, RPPActivationEvent, FutureConstraintEvent refs, and prior-tick RelationSedimentationEvent.
- It may make bounded updates to checking tendency, ambiguity tolerance, risk suspension, speech inhibition, threat sensitivity, or resentment pressure.
- It must not create personality labels directly.
- PersonView may later summarize these traces as apparent traits.
- every disposition change must cite the events that produced it.
- sedimented disposition contribution must decay unless reinforced, so it does not become a hidden fixed trait.
- Relation sediment may affect process tendencies only with a one-tick delay to avoid same-tick self-amplification.
```

## 10.1 Narrative Rendering Boundary

The viability layer may now be exposed to narrative rendering, but only as evidence.

It must not become a hidden plot engine.

Correct data flow:

```text
ViabilityRequirementEvent
+ ConstraintActivationEvent
+ AffordanceWidthEvent
+ DeformationTraceEvent
+ FutureConstraintEvent
+ RecognitionEvent
+ ActionSelectionEvent
+ ExpressionSelectionEvent
-> story frame viability summary
-> deterministic or LLM rendering
```

The rendering layer may translate viability evidence into:

```text
compressed action space
hesitation
silence
deflection
visible deformation
spatial pressure
timing pressure
objects carrying unresolved claims
```

The rendering layer may not translate viability evidence into:

```text
new motives
new biography
new relationship facts
new private decisions
new future predictions
new irreversible events
```

This is the key boundary:

```text
viability evidence explains why a moment is constrained.
It does not authorize what secretly "really happened."
```

If a rendered sentence cannot cite a lower event chain, it is literary invention and must be removed or marked as non-causal draft material.

## 10.2 Relation-Generated Positioning Boundary

`PositioningEvent` records how a relation temporarily places a process.

It is lower than character role and higher than raw event evidence.

Correct data flow:

```text
RecognitionEvent / RepairEvent / AvoidanceEvent / MisrecognitionEvent
+ NormativePressureEvent
+ FrameDefinitionEvent
+ RelevanceShiftEvent
+ RelationSedimentationEvent
+ RPPCompositionEvent
+ BindingUpdatedEvent
-> PositioningEvent
-> position_field.{process}.{position_type}
-> ActionSelectionEvent evidence position_*
-> RecognitionEvent evidence position_*
-> later derived PersonView / RelationshipView
```

Initial positions:

```text
claimant
debtor
defender
caretaker
controlled
public_performer
withdrawer
trapped_party
repair_partner
bound_party
```

These are not:

```text
personality traits
fixed social roles
desires
emotions
relationship labels
dramatic functions
```

They are relation-generated action positions.

The question is not:

```text
What kind of person is this?
```

The question is:

```text
What position has the relation made available, defensible, costly, shameful, or nearly unavoidable for this process?
```

This layer is required because the same process can move between contradictory positions without becoming an inconsistent character:

```text
claimant -> defender
debtor -> withdrawer
caretaker -> controlled
repair_partner -> trapped_party
public_performer -> claimant
```

Positioning must decay unless renewed.

It may bias action and recognition, but it must not directly cause a plot beat.

This prevents the simulator from smuggling "person" back into the ontology under softer names like role, archetype, social identity, or dramatic need.

## 11. Relationship to Dramatic Conflict Framework

The dramatic conflict framework is still useful, but it is not the bottom layer.

Its correct position is:

```text
Relational Viability Process Framework
-> Relation and Field Sedimentation
-> Conflict Evidence Kernel
-> Dramatic Conflict Views
-> Narrative Rendering
```

In other words:

```text
RPF does not simulate drama.
RPF simulates constrained relational viability.
Drama appears when constrained viability becomes observable, misrecognized, stabilized, and irreversible.
```

The conflict framework is therefore not the bottom upgrade path.

The bottom upgrade path is:

```text
field pressure
-> constraint activation
-> viability requirement
-> affordance narrowing
-> adaptive response / deformation
-> observation
-> recognition or misrecognition
-> memory reconstruction
-> future constraint
-> environment sedimentation
-> process disposition sedimentation
-> relation sedimentation
-> scheduler pressure / later constraint activation
-> affordance narrowing
-> recognition, repair, and fate selection
```

`RelationSedimentationEvent` is the bridge between local events and durable relation structure.

It lets the relation become causally real without turning "the relationship" into a primitive agent.

Relation sedimentation may produce:

```text
recognition debt
repair access narrowing
symbolic accounting load
future lock load
shared fate load
public definition load
asymmetry load
memory saturation
mutual predictability load
```

These are lower than dramatic conflict.

They explain why later conflict becomes likely, costly, unsayable, public, asymmetric, or irreversible.

The relation sedimentation feedback rule is:

```text
RelationSedimentationEvent
-> relation_sediment.* metric
-> ExpectationSedimentationEvent / expectation.* metric
-> AccountPressureEvent / account_pressure.* metric
-> NormativePressureEvent / norm_pressure.* metric
-> FrameDefinitionEvent / frame_definition.* metric
-> RelevanceShiftEvent / relevance_field.* metric
-> PositioningEvent / position_field.* metric
-> next-tick field and process disposition sedimentation
-> binding strength / exit-cost evolution
-> relation_sediment_load in scheduler preview
-> ConstraintActivationEvent(source_layer=relation_sedimentation)
-> ViabilityRequirementEvent with changed urgency, negotiability, or failure cost
-> AffordanceSelectionEvent evidence relation_*
-> AffordanceSelectionEvent evidence frame_*
-> AffordanceSelectionEvent evidence relevance_*
-> ActionSelectionEvent / ExpressionSelectionEvent evidence expected_*
-> ActionSelectionEvent / ExpressionSelectionEvent / RecognitionEvent evidence account_*
-> ActionSelectionEvent / ExpressionSelectionEvent / RecognitionEvent evidence norm_*
-> ActionSelectionEvent / RecognitionEvent evidence position_*
-> ExpressionSelectionEvent / RecognitionEvent evidence frame_*
-> RecognitionEvent evidence relation_*
-> FateTransition evidence relation_*
```

This is the minimum condition for saying that relation has become a causal process rather than a projected label.

```text

3. Conflict is not primitive.
```
